Requiring a network connection for the Windows Autopilot process
If you&rsquo;ve gone poking around in the available device configuration policies in Intune, you might have seen this one in the &ldquo;Device Restrictions&rdquo; category for &ldquo;Windows 10 and later&rdquo; devices, in the &ldquo;General&rdquo; section: But what does that &ldquo;Require users to connect to network during device setup&rdquo; actually do? The intent is simple: Don&rsquo;t let the user choose to skip making a network connection (since most machines are Wi-fi connected, they aren&rsquo;t connected by default) and therefore skipping the process of making it a corporate-joined (Azure AD or AD) and managed (Intune) device.&nbsp; (See the CSP documentation for the documentation for the behind-the-scenes MDM policy.)&nbsp; But the implementation of this is more complicated.&nbsp; In addition to this setting, you also need a second one, which is available in the Windows Autopilot profile: This &ldquo;Hide change account options&rdquo; setting keeps the user from saying &ldquo;I&rsquo;ll create a local account instead.&rdquo;&nbsp; (You might notice that the setting is disabled in this screenshot.&nbsp; It shouldn&rsquo;t be, we&rsquo;ll fix that.&nbsp; So right now, you can specify the setting when you create a new Autopilot profile, but you can&rsquo;t change the value after the profile is created.) So these two settings work together:&nbsp; One requires that the device connect to the network, which then makes sure the Autopilot profile is downloaded; the Autopilot profile setting then keeps the user from &ldquo;skipping out&rdquo; of the process.&nbsp; But these settings in practice are rather hard to implement: You can&rsquo;t get the &ldquo;Require users to connect to network during OOBE&rdquo; setting to the device before it is connected to a network, joined to AAD/AD, and enrolled in Intune &ndash; it&rsquo;s a device restriction policy delivered by Intune.&nbsp; (If you get a device that is configured in S Mode, this setting is in place by default.&nbsp; But due to compliance issues, this can&rsquo;t be enabled for non-S Mode installs.)You can&rsquo;t get the &ldquo;Hide change account options&rdquo; setting to apply if there&rsquo;s no network connection that can download the Autopilot profile.So, what good does it do?&nbsp; There are two scenarios that work well: White glove.&nbsp; If you have a technician pre-provision the device before it is delivered to the user, these settings can be applied and enforced so the user can&rsquo;t skip out.Reset/reimage.&nbsp; The &ldquo;Require users to connect to network during OOBE&rdquo; setting persists across Windows 10 installations.If you are paying attention, that second point should get your interest.&nbsp; How does a policy setting get persisted across Windows 10 installations?&nbsp; Simple, a UEFI variable is set to indicate that a network connection is required, and Windows 10 1809 and above always check that UEFI variable when going through OOBE.&nbsp; (If you need yet another reason to use UEFI, here it is.)&nbsp; And if your thoughts are jumping ahead to &ldquo;why can&rsquo;t the OEM set this variable before they ship us the devices&rdquo; (even when not using white glove) the answer is that they can &ndash; something we&rsquo;re looking into. There is an interesting side effect of this as well:&nbsp; If that UEFI variable is set and you try to apply a Windows 10 image using an unattend.xml, the device can get &ldquo;stuck&rdquo; in OOBE because it thinks you are trying to skip the network connection (and technically you are, using the unattend.xml to do it).&nbsp; So be careful with that if you are planning to do &ldquo;old-style&rdquo; reimaging of the device using MDT, ConfigMgr, etc.
